# Pay-Per-Call LLM API with Circle USDC Nano Payments

## Introduction

Every AI API on the internet today follows the same billing model:

```
Sign up  ──►  Enter credit card  ──►  Pick a plan  ──►  Billed monthly
```

That model works for humans. It completely breaks for AI agents.

An autonomous agent making thousands of API calls per session cannot sign up for subscriptions or hold credit cards. It needs to pay per action, programmatically, with no human in the loop. This is the gap that Circle's Gateway Nano Payments solve.

With Circle Gateway, the payment flow looks like this instead:

```
Fund wallet  ──►  Call the API  ──►  $0.01 USDC deducted  ──►  Response delivered
```

No accounts. No monthly invoices. No wallet pop-ups. The payment is embedded in the HTTP request itself.

In this tutorial you will build a working pay-per-call LLM API. A user (or agent) sends a message, the server demands $0.01 USDC, the buyer pays automatically using a gasless offchain signature, and the LLM responds. You will also build a React demo UI that visualizes the full payment flow in real time.

This project was built for the [Arc Agentic Economy Hackathon](https://lablab.ai/event) powered by Circle. The full source code is at [github.com/Stephen-Kimoi/nanopayments-with-usdc](https://github.com/Stephen-Kimoi/nanopayments-with-usdc).

### What you will build

- An Express server (the **seller**) that protects a `/chat` endpoint with Circle Gateway payment middleware. One line of code to require payment.
- A `GatewayClient` (the **buyer**) that automatically handles the 402 → sign → settle → response cycle
- A React + TypeScript demo UI showing the five-step payment pipeline animating in real time, with a receipt card after each successful call
- Two demo endpoints (`/demo/balance`, `/demo/chat`) so the UI can trigger real payments without exposing private keys in the browser

Let's get the project running before explaining how it works. Clone the repo, install dependencies, configure your environment, and start both the server and the UI.

### Clone and install

```bash
git clone https://github.com/Stephen-Kimoi/nanopayments-with-usdc.git
cd nanopayments-with-usdc
npm install
cd ui && npm install && cd ..
```

### Configure environment

Copy the example file and fill in your values:

```bash
cp .env.example .env
```

Open `.env` and set:

```bash
# Your wallet address — receives $0.01 USDC on each call
EVM_ADDRESS=0xYourSellerWalletAddressHere

# Your private key — the buyer wallet that pays per call.
# The server uses this internally; the browser never touches it.
```

### Deposit USDC into your Circle Gateway Wallet

Before the first payment can go through, the buyer wallet needs USDC deposited into the Circle Gateway Wallet. This is a one-time setup step. The Gateway Wallet is a smart contract at `0x0077777d7EBA4688BDeF3E311b846F25870A19B9` (same address on all EVM testnets) that holds your balance for gasless payments.

Per Circle's [Gateway documentation](https://developers.circle.com/stablecoins/gateway-overview), depositing funds moves USDC from your regular wallet into this contract so Circle can later settle signed payment authorizations on your behalf without requiring you to submit individual transactions.

The deposit script lives at [`client/deposit.ts`](https://github.com/Stephen-Kimoi/nanopayments-with-usdc/blob/main/client/deposit.ts). Here is what it does:

```typescript
// client/deposit.ts (lines 25–48)
// https://github.com/Stephen-Kimoi/nanopayments-with-usdc/blob/main/client/deposit.ts#L25-L48

const client = new GatewayClient({
  chain: "baseSepolia",
  privateKey: PRIVATE_KEY,
});

// Check your current balances before depositing
const before = await client.getBalances();
console.log(`Wallet USDC     : ${before.wallet.formatted} USDC`);
console.log(`Gateway balance : ${before.gateway.formattedAvailable} USDC`);

// client.deposit() handles two onchain transactions automatically:
//   1. Approve — allow the Gateway contract to spend your USDC
//   2. Deposit — move USDC from your wallet into the Gateway Wallet
const deposit = await client.deposit(depositAmount);
console.log(`Deposit tx: ${deposit.depositTxHash}`);

// Balance won't reflect immediately — Circle takes ~13-19 min on Base Sepolia
const after = await client.getBalances();
console.log(`New Gateway balance : ${after.gateway.formattedAvailable} USDC`);
```

`GatewayClient.deposit()` is the key call here. Under the hood it sends two transactions: an ERC-20 `approve` so the Gateway contract can pull your USDC, then a `deposit` call into the contract. Once the deposit confirms onchain, Circle picks it up and makes the balance available for gasless payments.

Run the deposit:

```bash
npm run deposit -- 0.10
```

Base Sepolia takes **13 to 19 minutes** for Circle to credit the deposit after the onchain transaction confirms. Poll until the balance appears:

```bash
npm run balance
```

The balance polling script is at [`client/check-balance.ts`](https://github.com/Stephen-Kimoi/nanopayments-with-usdc/blob/main/client/check-balance.ts#L14-L19). It calls `client.getBalances()` every 60 seconds and exits when the Gateway balance goes above zero.

When you see the Gateway balance go above 0, you are ready.

## How the Demo Works

When you hit Send in the UI, here is exactly what happens:

**Step 1: Request sent.** The UI calls `POST /demo/chat` on the Express server. This is a demo-only endpoint that acts as the buyer on your behalf, so the browser never handles private keys.

**Step 2: 402 Payment Required.** Inside `/demo/chat`, the server's embedded `GatewayClient` calls the real `POST /chat` endpoint. The Circle Gateway middleware sees no payment proof and returns HTTP 402 with machine-readable payment requirements: the price, the network, the payment scheme.

**Step 3: Sign authorization.** The `GatewayClient` reads the 402 response and creates an EIP-3009 `TransferWithAuthorization` signature. This is a cryptographic message that authorizes Circle to move $0.01 USDC from the buyer's Gateway Wallet to the seller. This happens **offchain**: no blockchain transaction, no gas fee.

**Step 4: Circle Gateway settle.** The client retries the request with the signature attached. The middleware forwards it to Circle's settle API, which validates the signature and queues the transfer. Circle batches these offchain signatures and settles them onchain periodically. This is why there is no gas per call.

**Step 5: Response delivered.** The server calls OpenAI, gets the LLM response, and returns it along with a payment receipt: amount paid, network, scheme, and balance before and after.

The whole cycle from the first request to the LLM response takes about two seconds.

## The Code Explained

### Seller side: one line to require payment

The server is a standard Express app with Circle's Gateway middleware. See the full file at [`server/index.ts`](https://github.com/Stephen-Kimoi/nanopayments-with-usdc/blob/main/server/index.ts).

The middleware is set up at [lines 35–38](https://github.com/Stephen-Kimoi/nanopayments-with-usdc/blob/main/server/index.ts#L35-L38):

```typescript
// server/index.ts (lines 1–38)
// https://github.com/Stephen-Kimoi/nanopayments-with-usdc/blob/main/server/index.ts#L1-L38

import { createGatewayMiddleware } from "@circle-fin/x402-batching/server";

const gateway = createGatewayMiddleware({
  sellerAddress: SELLER_ADDRESS,  // your receiving wallet
  networks: ["eip155:84532"],     // Base Sepolia
});
```

The protected route is at [lines 118–142](https://github.com/Stephen-Kimoi/nanopayments-with-usdc/blob/main/server/index.ts#L118-L142):

```typescript
// server/index.ts (lines 118–142)
// https://github.com/Stephen-Kimoi/nanopayments-with-usdc/blob/main/server/index.ts#L118-L142

app.post("/chat", gateway.require("$0.01"), async (req, res) => {
  // req.payment is populated by the middleware on success
  const { payer, amount, network } = (req as any).payment;
  console.log(`Payment received: ${amount} USDC from ${payer} on ${network}`);

  const completion = await openai.chat.completions.create({
    model: MODEL,
    messages: req.body.messages,
  });

  res.json({
    reply: completion.choices[0].message.content,
    model: completion.model,
    usage: completion.usage,
  });
});
```

`gateway.require("$0.01")` is the entire payment gate. If the request arrives without a valid payment proof, the middleware returns 402 automatically. If the payment is valid, it populates `req.payment` and calls `next()`. Your route handler only runs after the payment has cleared.

### Buyer side: automatic payment handling

The standalone buyer client is at [`client/pay.ts`](https://github.com/Stephen-Kimoi/nanopayments-with-usdc/blob/main/client/pay.ts). The key section is [lines 26–68](https://github.com/Stephen-Kimoi/nanopayments-with-usdc/blob/main/client/pay.ts#L26-L68):

```typescript
// client/pay.ts (lines 26–68)
// https://github.com/Stephen-Kimoi/nanopayments-with-usdc/blob/main/client/pay.ts#L26-L68

const client = new GatewayClient({
  chain: "baseSepolia",
  privateKey: PRIVATE_KEY,
});

// Check balance before attempting payment
const balances = await client.getBalances();
console.log(`Gateway balance : ${balances.gateway.formattedAvailable} USDC`);

if (balances.gateway.available < 10_000n) {
  console.log("Gateway balance too low. Run: npm run deposit");
  process.exit(1);
}

// GatewayClient.pay() handles the full x402 flow automatically:
//   send request → receive 402 → sign EIP-3009 → retry → get response
const { data, status } = await client.pay(url, {
  method: "POST",
  body: JSON.stringify({ messages }),
  headers: { "Content-Type": "application/json" },
});

console.log(`Status : ${status}`);
console.log(`Reply  : ${data.reply}`);
```

The caller doesn't need to know anything about payment. `pay()` handles the 402 cycle transparently and returns the final response.

The private key normalization (handling wallets that export keys without `0x`) happens at [lines 17–19](https://github.com/Stephen-Kimoi/nanopayments-with-usdc/blob/main/client/pay.ts#L17-L19):

```typescript
// client/pay.ts (lines 17–19)
const rawKey = process.env.PRIVATE_KEY || process.env.EVM_PRIVATE_KEY || "";
const PRIVATE_KEY = (rawKey.startsWith("0x") ? rawKey : `0x${rawKey}`) as `0x${string}`;
```

### Demo endpoints for the UI

The UI calls `/demo/balance` and `/demo/chat`. These are wrapper endpoints on the same server that embed a `GatewayClient` instance, so the browser never needs a private key.

The embedded buyer client is initialized at [lines 18–26](https://github.com/Stephen-Kimoi/nanopayments-with-usdc/blob/main/server/index.ts#L18-L26):

```typescript
// server/index.ts (lines 18–26)
// https://github.com/Stephen-Kimoi/nanopayments-with-usdc/blob/main/server/index.ts#L18-L26

const rawKey = process.env.PRIVATE_KEY || process.env.EVM_PRIVATE_KEY || "";
const buyerClient = rawKey
  ? new GatewayClient({
      chain: "baseSepolia",
      privateKey: (rawKey.startsWith("0x") ? rawKey : `0x${rawKey}`) as `0x${string}`,
    })
  : null;
```

The `/demo/balance` endpoint is at [lines 42–57](https://github.com/Stephen-Kimoi/nanopayments-with-usdc/blob/main/server/index.ts#L42-L57) and the `/demo/chat` endpoint at [lines 59–105](https://github.com/Stephen-Kimoi/nanopayments-with-usdc/blob/main/server/index.ts#L59-L105):

```typescript
// server/index.ts (lines 42–105)
// https://github.com/Stephen-Kimoi/nanopayments-with-usdc/blob/main/server/index.ts#L42-L105

// GET /demo/balance: returns live gateway + wallet balances for the header
app.get("/demo/balance", async (_req, res) => {
  const b = await buyerClient.getBalances();
  res.json({
    gateway: b.gateway.formattedAvailable,
    wallet: b.wallet.formatted,
  });
});

// POST /demo/chat: runs the full payment cycle server-side
app.post("/demo/chat", async (req, res) => {
  const before = await buyerClient.getBalances();
  const balanceBefore = before.gateway.formattedAvailable;

  // pay() calls the real /chat endpoint, which requires $0.01 USDC
  const { data } = await buyerClient.pay(`${SERVER_URL}/chat`, {
    method: "POST",
    body: JSON.stringify({ messages: req.body.messages }),
    headers: { "Content-Type": "application/json" },
  });

  const after = await buyerClient.getBalances();

  res.json({
    reply: data.reply,
    model: data.model,
    usage: data.usage,
    payment: {
      amount: "0.01",
      currency: "USDC",
      network: "Base Sepolia",
      scheme: "GatewayWalletBatched",
      balanceBefore,
      balanceAfter: after.gateway.formattedAvailable,
    },
  });
});
```

## How an AI Agent Would Actually Use This

The seller code above could be deployed to any public URL, for example `https://nanoai.example.com/chat`. An AI agent that needs to call it would run exactly what is in [`client/pay.ts`](https://github.com/Stephen-Kimoi/nanopayments-with-usdc/blob/main/client/pay.ts):

```typescript
const client = new GatewayClient({
  chain: "baseSepolia",
  privateKey: AGENT_PRIVATE_KEY,   // the agent's funded wallet
});

const { data } = await client.pay("https://nanoai.example.com/chat", {
  method: "POST",
  body: JSON.stringify({
    messages: [{ role: "user", content: agentQuery }],
  }),
  headers: { "Content-Type": "application/json" },
});
```

The agent doesn't sign up for an account, doesn't hold an API key, and doesn't have a subscription. It has USDC in a Gateway Wallet and pays per call. The operator of the AI service receives USDC in real time, per request, with no billing infrastructure to maintain.

An orchestrator framework like LangChain, AutoGen, or a custom agent loop would wrap all outbound HTTP calls with `GatewayClient.pay()`. When the agent picks "call the LLM tool," the payment happens transparently inside the tool implementation. The agent never needs to know payment is involved.

## Why This Matters: The Economics of Nano Payments

Traditional payment rails carry a minimum viable transaction cost of roughly $0.30 (Stripe's fixed fee alone). That floor makes anything under about $1 per call economically unworkable — you would pay more in fees than the service is worth. So every AI API today is forced into subscriptions or credit bundles.

Circle Gateway removes that floor entirely:

| | Traditional API billing | Circle Gateway Nano Payments |
|---|---|---|
| Minimum viable price | ~$1+ per call | $0.001 per call |
| Gas per payment | N/A (card fees apply) | $0.00 |
| Human required to set up | Yes (account, card, plan) | No |
| Autonomous agent compatible | No | Yes |
| Payment settled | Monthly invoice | Near real-time, batched onchain |

The $0.01/call price in this project is arbitrary. You could charge $0.001, $0.005, or $0.10. The payment mechanics work at any of those price points. For an agent making 10,000 calls per day, the difference between a flat monthly subscription and paying exactly for what you use is enormous, especially when usage patterns are unpredictable.

## The Payment Scheme: GatewayWalletBatched

Circle's scheme is called **GatewayWalletBatched**. It is built on [EIP-3009](https://eips.ethereum.org/EIPS/eip-3009) (`transferWithAuthorization`), which allows an authorized party to transfer tokens on a holder's behalf using a cryptographic signature, without the holder submitting a transaction.

The flow in detail:

1. **Deposit** — the buyer deposits USDC into the Gateway Wallet smart contract (`0x0077777d7EBA4688BDeF3E311b846F25870A19B9` on all EVM testnets). See [`client/deposit.ts`](https://github.com/Stephen-Kimoi/nanopayments-with-usdc/blob/main/client/deposit.ts).
2. **Sign** — for each payment, the buyer signs an EIP-3009 message offchain: "I authorize Circle to transfer $X from my gateway balance to seller Y, valid between timestamp A and B."
3. **Submit** — Circle collects these signatures and settles them onchain in batches. Each individual call costs zero gas because it is the batch that goes onchain, not each individual transfer.
4. **Receive** — the seller's address receives USDC onchain, net of any Circle fees.

The key insight is that the signature is the payment proof. The middleware can verify a signature instantly without waiting for a blockchain transaction. Settlement happens asynchronously in the background.

## Common Issues

**Gateway balance shows 0 after deposit.** Base Sepolia takes 13 to 19 minutes for Circle to credit a deposit after the onchain transaction confirms. Run `npm run balance` to poll every 60 seconds until it appears. See [`client/check-balance.ts`](https://github.com/Stephen-Kimoi/nanopayments-with-usdc/blob/main/client/check-balance.ts).

**"PRIVATE_KEY is required" error.** Make sure your `.env` has `EVM_PRIVATE_KEY` set. The client reads `PRIVATE_KEY` or `EVM_PRIVATE_KEY` and normalizes it to include the `0x` prefix automatically. See [lines 17–19 of `client/pay.ts`](https://github.com/Stephen-Kimoi/nanopayments-with-usdc/blob/main/client/pay.ts#L17-L19).

**Server returns 402 repeatedly.** The client's Gateway balance is 0 or below the $0.01 threshold. Deposit more USDC and wait for Circle to credit it.

**`/demo/chat` returns 503.** The server could not find a buyer private key in the environment. Check that `EVM_PRIVATE_KEY` is set and restart the server after editing `.env`. See [lines 18–26 of `server/index.ts`](https://github.com/Stephen-Kimoi/nanopayments-with-usdc/blob/main/server/index.ts#L18-L26).

## Next Steps

- **Deploy the server** to a public URL (Railway, Render, or a VPS). Any agent with a funded Gateway Wallet can use it immediately, no sign-up required.
- **Change the price.** Try `gateway.require("$0.001")` for sub-cent calls, or `gateway.require("$0.10")` for heavier models.
